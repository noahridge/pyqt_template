HMI Template — Architecture Guide

A robust, scalable PyQt6 foundation for human-machine interface applications.

This document explains how the template is structured and, more importantly, why each decision was made. It is aimed at HMI-style applications — industrial control panels, instrument front-ends, plant dashboards — that must run unattended for long periods without crashing, freezing, or hiding faults.

The guiding principle throughout: a failure in one part of the UI must never take down the whole panel, and nothing slow may ever run on the GUI thread. Everything below follows from those two rules.

2. The robustness problem

A default PyQt application has three silent ways to fail on a 24/7 panel. The template exists largely to close all three:

Key fact. Since PyQt 5.5, an unhandled exception inside a slot calls sys.excepthook and then aborts the process. This is the single most important reason HMI code needs the @safe_slot guard and a global fault hook.

3. Layered architecture

The codebase is split into strict layers whose dependencies all point inward, toward a stable core. A layer may use the layers below it but never the layers above — so core knows nothing about views, and the domain knows nothing about widgets. This is what lets the same foundation host a 3-screen demo or a 50-screen plant HMI.

Why this matters: cross-layer communication never happens through direct references — it flows through a SignalBus (§9). A new page subscribes to events without any service knowing it exists. That decoupling is the difference between a project that scales and one that turns into a web of mutual imports.

4. Runtime data flow

Data moves in one direction. A device reading travels device → service → tag store → bus → views, and operator intent travels back as explicit commands. Unidirectional flow makes a large UI predictable: there is exactly one place a value can change (the store) and exactly one way views learn about it (the bus).

5. Fault handling core

Every unexpected error — from a slot, a Qt worker, a plain Python thread, or Qt's C++ layer — is converted into one immutable Fault object and broadcast on a single FaultBus. Because the bus is a QObject, a fault raised on any thread is delivered as a queued event on the GUI thread, where a status bar or dialog can react safely.

Installing the hooks is a single call made before the event loop starts:

def install_fault_hooks() -> None:
    """Install all global hooks. Call once, early, before the event loop starts."""
    sys.excepthook = _python_excepthook          # escaped from a slot
    threading.excepthook = _threading_excepthook  # escaped from a thread
    _install_qt_message_handler()                 # Qt's own C++ messages

The Fault carries enough to decide how loud to be. An AppError subclass declares its own severity and whether it is recoverable, so the UI treats a dropped device link (recoverable, status bar) differently from a config error (critical, modal dialog):

@dataclass(frozen=True)
class Fault:
    message: str
    severity: Severity = Severity.ERROR
    source: str = "app"
    exception: BaseException | None = None
    recoverable: bool = True

6. Safe slots core

The hooks in §5 are the safety net; @safe_slot is the everyday guard you actually reach for. It wraps a slot so any exception is reported through the fault system and swallowed — the event loop keeps spinning. Decorate every method you connect to a signal.

class Dashboard(QWidget):
    @safe_slot
    def on_refresh_clicked(self):
        ...                       # may raise; the app stays alive

    @safe_slot(default=0.0, source="dashboard")
    def compute(self) -> float:
        ...                       # returns 0.0 if it raises

The implementation is deliberately small — a try/except that routes to report_fault and returns a default instead of propagating:

@functools.wraps(fn)
def wrapper(*args, **kwargs):
    try:
        return fn(*args, **kwargs)
    except Exception as exc:          # deliberate catch-all
        label = source or _derive_source(fn, args)
        report_fault(exc, source=label, severity=severity)
        if reraise:
            raise
        return default
Why a catch-all is correct here. Catching bare Exception is normally a smell, but a slot is a boundary between Qt's C++ event loop and your Python code. An exception that crosses that boundary aborts the process — so catching everything at that one boundary and converting it to a fault is exactly right. Combine with @pyqtSlot (outermost) if you also want the typed C++ slot.

7. Threading & workers core

HMI apps spend most of their time talking to slow, unreliable things. All of that must happen off the GUI thread, or the UI freezes and the operator can't hit the stop button. The template uses the worker-object pattern Qt recommends — a QObject moved onto a QThread — rather than subclassing QThread.

WorkerThread owns the worker + thread pair and guarantees an orderly shutdown. The subtle part is stopping cleanly: a QTimer created on the worker thread must be stopped on that same thread, or Qt raises QObject::killTimer: Timers cannot be stopped from another thread. The fix is a blocking-queued call so the worker tears down its own thread-affine resources before the loop quits:

def stop(self, *, wait_ms: int = 5000) -> None:
    if not self._thread.isRunning():
        return
    # Run stop() ON the worker thread and block until it's done, so timers/
    # sockets are torn down on the correct thread before we quit the loop.
    if hasattr(self._worker, "stop"):
        QMetaObject.invokeMethod(
            self._worker, "stop", Qt.ConnectionType.BlockingQueuedConnection)
    self._thread.quit()
    if not self._thread.wait(wait_ms):
        self._thread.terminate()      # last resort
        self._thread.wait()
Lesson baked into the template. Thread affinity is the No.1 source of hard crashes in Qt apps. The rule the code enforces: an object that owns a timer, socket, or file handle must create and destroy it on the same thread. PeriodicWorker creates its QTimer in run() and stops it in stop() — both on the worker thread.

A failing poll is reported but does not stop the loop — transient device errors must not end polling:

def _tick(self):
    try:
        self.step()                  # your driver call; may raise DeviceError
    except Exception as exc:        # keep polling alive
        fault = report_fault(exc, source=self._name)
        self.failed.emit(fault)

8. Connection state machine core

Every external connection moves through one explicit state machine, owned by the Service base class. The example DeviceService translates a run of poll failures into degraded and eventually faulted, and recovers automatically — the status indicator in the UI simply binds to these states.

9. Model & signal bus core

9.1 What a tag is

A tag is the atom of an HMI's data model: one named process value with its metadata. In SCADA/HMI terminology a tag is a tank level, a motor speed, a valve state — anything the system reads from or writes to the field. The template models it as a small dataclass that couples the value with the three things an operator actually needs alongside it:

@dataclass
class Tag:
    name: str
    value: Any = None
    unit: str = ""
    quality: Quality = Quality.UNKNOWN     # is this value trustworthy?
    timestamp: datetime = ...              # when was it last good?
    low_limit: float | None = None          # engineering limits → alarms
    high_limit: float | None = None

Bundling value + quality + timestamp + limits is the whole point. A bare float cannot tell an operator the difference between "the tank really is at 0%" and "we lost contact and 0 is just the last number we saw." On a panel controlling real equipment, that distinction is a safety issue — so it is encoded in the model, not left to chance.

9.2 Quality — the most important field

Quality is what makes the store trustworthy. The UI styles a value differently depending on it, so stale or invalid data is never silently presented as live:

QualityMeaningTypical UI treatment
GOODFresh, trustworthy reading.Show the value normally.
STALEConnection lost; this is the last known value.Grey it out / show its age.
BADDevice reported the value as invalid.Fault styling; don't trust it.
UNKNOWNNever read yet (startup).Show —.

When the device service drops to FAULTED (§8), it calls one method to flag the whole dataset at once — the UI immediately reflects that data is no longer live, without waiting for individual updates that will never arrive:

def mark_stale(self):           # called by the service on disconnect
    for tag in self._tags.values():
        tag.quality = Quality.STALE

9.3 Single source of truth: one writer, many readers

The TagStore is an in-memory registry of current tag values, and it is the only place a value lives. Services write to it; everything else reads from it and subscribes to its change notifications. No widget ever talks to a device. This single-writer / many-reader shape is what keeps a large UI consistent — there is exactly one source of a value and exactly one way to learn it changed.

9.4 The update lifecycle

update() is the one entry point for changing a value. It upserts the tag (auto-defining unknown ones so services can publish freely), stamps quality and time, and broadcasts the change:

def update(self, name, value, *, quality=Quality.GOOD) -> Tag:
    tag = self._tags.get(name) or Tag(name=name)
    tag.value, tag.quality, tag.timestamp = value, quality, datetime.now()
    self._tags[name] = tag
    bus().tagUpdated.emit(name, value)   # every reader hears it
    return tag

Traced end to end, a single reading flows like this:

1. worker reads the device        # on the WORKER thread
2. emits reading(name, value)     # queued across the thread boundary
3. service slot → store.update()  # now on the GUI thread
4. store writes the Tag, quality=GOOD, timestamp=now
5. store emits bus().tagUpdated   # GUI thread
6. Dashboard, Control VM, AlarmManager, … all react   # GUI thread

9.5 Why it needs no locks

Thread-safety by design, not by mutex. The store is plain Python with no locks — and that's correct, not a bug. The worker→service signal in step 2 is queued, so update() always runs on the GUI thread. With a single writer thread there is no race to guard. The thread boundary is crossed exactly once, by Qt, in a place that is safe by construction. If you ever need to write from several threads, keep funnelling through queued signals rather than reaching for a QMutex — it stays simpler and faster.

9.6 Scaling the store

A dict-backed store is ideal up to thousands of tags read by ordinary widgets. For very large tabular displays (e.g. a 10,000-row tag grid), back the store with a QAbstractTableModel so Qt's view classes can render and update only what's visible. The data-flow and bus pattern around it stay exactly the same — only the storage and the readers' binding change.

9.7 The signal bus

The SignalBus is a process-wide QObject singleton holding only genuinely cross-cutting events. Producers emit; consumers connect; neither knows the other exists.

class SignalBus(QObject):
    navigationRequested   = pyqtSignal(str)            # switch page by key
    connectionStateChanged = pyqtSignal(str, object)    # (service, state)
    tagUpdated            = pyqtSignal(str, object)     # (tag, value)
    alarmRaised           = pyqtSignal(object)
    alarmCleared          = pyqtSignal(object)
Discipline note. The bus is for cross-cutting events only. Parent-child or single-screen communication should use direct signals — overusing a global bus makes flow as hard to trace as global variables.

10. Alarm engine core

Alarms are first-class in HMI: they drive operator attention and event logs. The AlarmManager subscribes to tagUpdated, compares values against the high/low limits declared on each tag, and emits alarmRaised / alarmCleared — latching until the value returns in range. The alarms page and status bar just listen.

def _evaluate(self, name, value, alarm_type, limit):
    if limit is None: return
    in_alarm = value > limit if alarm_type is AlarmType.HIGH else value < limit
    key = f"{name}:{alarm_type.value}"
    if in_alarm and key not in self._active:
        self._active[key] = alarm = Alarm(name, alarm_type, limit, value)
        bus().alarmRaised.emit(alarm)
    elif not in_alarm and key in self._active:
        bus().alarmCleared.emit(self._active.pop(key))

11. Views & navigation

The MainWindow is a shell — side-nav + a Navigator (a router over QStackedWidget) + a status bar. It owns no business logic. Pages subclass BasePage, which gives them on_show / on_hide lifecycle hooks so expensive work (chart animations, fast polling) only runs while a page is visible — important on constrained HMI hardware.

Navigation is decoupled: anything can request a screen change by emitting on the bus, so an alarm banner can jump to the Alarms page without holding a reference to it:

bus().navigationRequested.emit("alarms")   # from anywhere; Navigator handles it

12. Optional MVVM opt-in

The template is not MVVM by default. Most HMI screens are thin read-outs with no logic worth isolating, so they bind to the model directly. But screens with real presentation logic — commands, validation, preconditions, derived state — benefit from a ViewModel that holds that logic out of the widget so it can be unit-tested without any UI.

RoleResponsibilityHolds a reference to…
Modeldata + domain rules (TagStore, services)nothing above it
ViewModelpresentation logic: bindable state + commandsthe model
Viewrender VM state, forward clicks; UI concerns onlyits ViewModel

The example is a pump-control screen. The View clicks call commands; the VM checks preconditions and either acts or rejects with a reason. Crucially, the VM holds no reference to the view.

Bindable state is a pyqtProperty with a notify signal; a command is a pyqtSlot that validates before acting:

@pyqtProperty(bool, notify=canStartChanged)
def can_start(self) -> bool:
    return self._check_preconditions()[0]

@pyqtSlot()
def start(self) -> None:
    ok, reason = self._check_preconditions()      # device connected? value fresh?
    if not ok:                                   # tank below high limit?
        self.commandRejected.emit(reason); return
    self._running = True                          # a real impl writes to the device
    self._emit_all()
    self.commandAccepted.emit("Pump started")

Widget MVVM's one real cost is manual binding (PyQt has no declarative binding like QML). The bind() helper reduces it to one line per field:

bind(vm.statusChanged,   lambda: vm.status_text, label.setText)
bind(vm.canStartChanged, lambda: vm.can_start,   start_button.setEnabled)
The payoff, verified. Because all the rules live in the VM, tests/test_viewmodel.py exercises "blocked when disconnected", "refuses above high limit", "starts when ready", etc. — with no widget instantiated. That testability is the entire reason to add a ViewModel.

It's opt-in. The Control page is registered only when enable_control_demo is true. Set it false — or delete views/viewmodels/ and control_page.py — to drop MVVM entirely. MVVM and the bus are complementary: the bus is how the VM receives model updates.

13. Startup sequence

Order is deliberate: each step makes the next observable or safe. app.py is the only place that knows the concrete object graph.

# hmi/app.py — main()
1. configure_logging()       # so everything after is recorded
2. install_fault_hooks()     # so any later failure is captured, not fatal
3. AppConfig.load()          # defaults < JSON < env, then validated
4. TagStore + define tags    # the data model & alarm limits
5. DeviceService, AlarmMgr   # background I/O & rules (not started yet)
6. MainWindow + pages        # UI built, indicators wired to the bus
7. window.show()
8. device.start()            # I/O begins only once the UI can receive signals
9. app.exec()                # event loop; device.stop() in finally

14. Rationale at a glance

DecisionWhyAlternative rejected
Global fault hooks + FaultBus An HMI must survive a bad slot/thread; faults must reach a log and the operator. Let exceptions abort (default) — unacceptable on a 24/7 panel.
@safe_slot on every slot The slot boundary is where exceptions become fatal; guard them there. Hope no slot ever raises.
Worker-object on a QThread Qt-recommended; keeps I/O off the GUI thread; clean signal interface. Subclass QThread — easy to get affinity wrong.
Blocking-queued stop() Thread-affine resources (timers) must die on their own thread. Quit the loop directly — causes killTimer crashes.
Single SignalBus Decouples layers; new screens subscribe without touching producers. Direct references everywhere — import spaghetti at scale.
TagStore as single source of truth One writer, many readers → a consistent UI; views never touch devices. Each widget polls the device — races and frozen UI.
Explicit connection state machine Operators need to know connected vs degraded vs faulted, with auto-recovery. A boolean "connected" flag — hides degradation.
MVVM optional, per-screen VMs pay off only where there's logic to test; read-outs don't need the boilerplate. MVVM everywhere (empty VMs) or never (untestable logic in widgets).
Typed AppConfig (defaults < JSON < env) One image, many panels; per-station overrides without code changes. Constants in code, or untyped globals.

15. How to extend

GoalDo this
Add a screenSubclass BasePage, build UI in build_ui(), register with window.add_page(...) in app.py.
Add a real deviceSubclass Service + a PeriodicWorker; in step() call your driver and raise DeviceError on failure. State machine & threading come free.
Add an alarmGive a Tag a high_limit/low_limit when you define it. The engine does the rest.
Add a control screenWrite a ViewModel with pyqtProperty state + pyqtSlot commands; bind a thin View to it; unit-test the VM.
React to an app-wide eventConnect to a signal on bus() — don't thread references through constructors.
Write a crash-proof slotDecorate it with @safe_slot (and @pyqtSlot outermost if you want the typed slot).
